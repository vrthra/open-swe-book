# Chapter 1 — Exercises

Exercises are graded by depth: **[warm‑up]** checks understanding, **[analysis]** asks
you to reason, **[project]** connects to your team project (Appendix A).

## Concepts

1. **[warm‑up]** In your own words, give one difference between *programming* and
   *software engineering*, and one example task that belongs to each.
2. **[warm‑up]** Define *mistake*, *fault*, and *failure*, and give a single concrete
   scenario (a few sentences of a story) that contains all three in sequence.
3. **[warm‑up]** Distinguish *essential* from *accidental* complexity. Give one example
   of each from a system you have used or built.

## Analysis

4. **[analysis]** The "iron triangle" says scope, cost, and time are coupled. A manager
   says: "Add these five features (more scope), ship two weeks earlier (less time), and
   don't hire anyone (same cost)." Explain, using the triangle, why at most one and a
   half of these is achievable, and propose two honest options you could offer instead.
5. **[analysis]** Agile methods "fix time and cost and let scope flex." Describe a kind
   of project where that trade‑off is *wrong* — where scope must be fixed and time should
   flex instead — and justify it.
6. **[analysis]** Re‑read the two‑companies story (§1.1.2). Identify the specific moment
   at which Company A's velocity begins to collapse, and name the two engineering
   disciplines from later chapters that would most have helped, and why.
7. **[analysis]** Black‑box testing "stays valid when the code is rewritten," while
   white‑box testing "finds code the spec never reaches." Construct a small example
   function where a black‑box suite would pass but a serious fault remains that only
   white‑box testing would catch.

## Ethics

8. **[analysis]** The Volkswagen defeat device "worked exactly as designed." Explain why
   better testing could not have prevented this scandal, and which principle of the
   ACM/IEEE code was most directly violated.
9. **[analysis]** List the *defense‑in‑depth* failures in the Therac‑25 case. For each,
   name a safeguard (from anywhere in this book) that could have provided a second layer.
10. **[analysis]** You are asked to implement a feature you believe misleads users but is
    legal. Using the ACM code as your basis, script (in ~150 words) how you would raise
    the concern with your team and manager.

## Project

11. **[project]** For your team‑project idea, list every distinct *stakeholder* (end
    users, secondary users, sponsors, regulators). For each, write one sentence on what
    they most need and one risk if that need is missed.
12. **[project]** Name one requirement of your project that you expect to *change* during
    the term. Describe one *process‑level* and one *design‑level* way you will make that
    change cheap to accommodate.
13. **[project]** Identify one way your project could cause real harm if it failed
    (privacy, safety, money, fairness). Write a one‑paragraph "responsibility note" for
    your team's README describing how you will guard against it.
